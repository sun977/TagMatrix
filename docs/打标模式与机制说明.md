# TagMatrix 打标模式与执行策略机制说明

本文档详细说明了 TagMatrix 系统中打标任务的核心机制，包括“执行策略”与“打标模式”的区别，以及相关的核心代码实现位置。主要供后续系统迭代和扩展（如引入主标签优先算法）参考。

## 1. 核心概念区分

系统现已将对历史数据的处理逻辑与新标签生成逻辑拆分为两个独立的概念：

### 1.1 执行策略 (Execution Strategy)
控制当一条数据重新执行打标任务时，如何处理它**原有的自动标签**。
- **追加模式 (append)**：保留该数据原有的标签，直接追加写入本次命中的新标签。
- **覆盖模式 (overwrite)**：在写入新标签前，先**批量清除**该数据原有的、由打标引擎自动打上的标签（`source = 'auto_rule'`）。

### 1.2 打标模式 (Tagging Mode)
控制当一条数据在本次打标任务中**同时命中多条规则**时，如何生成并分配最终的标签。
- **多标签模式 (multiple)**：默认行为。所有命中的规则都会生成对应的标签，且地位同等（`is_primary = false`）。
- **单标签模式 (single)**：即使命中多条规则，也仅取优先级最高的**一个**标签入库。
- **混合模式 (mixed)**：所有命中的规则都会生成标签，但**优先级最高**的那一个标签会被设为主标签（`is_primary = true`），其余同等为次标签。

---

## 2. 当前排序机制与兜底逻辑

在“单标签模式”与“混合模式”下，系统需要决定哪一个标签是“最优先的”。当前采用的排序依据如下：

1. **Priority (优先级) 比较**：优先比较规则表中配置的 `Priority` 字段，数值越大优先级越高。
2. **ID 比较 (Tie-breaker 兜底)**：当 `Priority` 发生相同时，比较规则的 `ID`，ID 越大（说明创建/修改时间越晚）的规则优先胜出。

## 3. 代码位置标注 (后续算法扩展点)

相关的处理逻辑集中在后端引擎的核心调度文件中。如果您后续需要引入复杂的**主标签优先算法**（如：基于机器学习动态打分、特定业务线标签权重覆盖等），请直接在以下位置进行修改。

**核心文件路径**: `internal/service/taskengine/taskengine_service.go`
**核心方法**: `processRecords`

### 3.1 规则排序算法扩展位置
系统会将当前数据命中的所有规则收集到 `matchedRules` 数组中，随后进行排序。
**后续在此处替换为您的高级优先算法**：

```go
// 代码位置: processRecords 方法内部
// 查找关键词: "按规则优先级降序排序" 或 "sort.SliceStable"

// 【扩展点】：在此处接入主标签优先算法
sort.SliceStable(matchedRules, func(i, j int) bool {
    // 默认逻辑: 比较 Priority
    if matchedRules[i].model.Priority != matchedRules[j].model.Priority {
        return matchedRules[i].model.Priority > matchedRules[j].model.Priority
    }
    // 默认兜底逻辑: 比较 ID
    return matchedRules[i].model.ID > matchedRules[j].model.ID
})
```

### 3.2 模式应用逻辑
只要在上面的 `sort.SliceStable` 中保证了**最优的规则排在数组的第一位**（即 `matchedRules[0]`），下方的模式逻辑会自动生效，**无需修改**：

```go
// 代码位置: processRecords 方法内部，紧跟在排序逻辑下方

// 单标签模式：自动截取排序后的第一项
if tagMode == "single" {
    matchedRules = matchedRules[:1]
}

for i, pr := range matchedRules {
    isPrimary := false
    // 混合模式：自动将排在第一项的规则标记为主标签
    if tagMode == "mixed" && i == 0 {
        isPrimary = true
    }
    // ... 执行生成 SysEntityTag 并入库
}
```
