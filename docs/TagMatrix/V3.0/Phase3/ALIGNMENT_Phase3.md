# 需求对齐文档 - Phase 3: 打标任务引擎与看板改造

## 原始需求
根据 `V3.0_REFACTORING_PLAN.md` 执行 Phase 3 重构：
*   **后端**：Matcher 引擎在执行打标任务时，严格按照传入的 `dataset_id` 去 `sys_match_rules` 中筛选专属规则，并只在归属于该 `dataset_id` 的原始数据中进行运算。
*   **前端**：【打标任务看板】发起打标前，增加“选择目标数据集”的必填项。看板数据展示时，支持按数据集过滤和预览。

## 项目上下文

### 现有进展
- 经代码审查，**核心引擎的规则与数据隔离（按 `dataset_id`）** 已在 Phase 2 同步提前完成（位于 `taskengine_service.go` 的 `RunTaggingTask` 中）。
- **前端发起打标前的“选择目标数据集”** 必填项也已在 `TaskKanban.vue` 的创建表单中实现。

### 遗留缺口（Phase 3 真实工作量）
1.  **看板历史任务的列表展示**：`TaskKanban.vue` 的“任务历史”表格缺少 `目标数据集` 的展示列，且无法按数据集进行过滤。
2.  **后端接口调整**：后端 `GetTaskBatches` 接口目前仅返回底层 `TagTaskBatch` 实体，前端仅拿到 `dataset_id`，缺乏展示用的 `dataset_name`。需要引入 DTO 或前端组装映射。
3.  **预览功能验证**：需要核实看板内的数据预览逻辑是否符合“按数据集预览”的需求（打标看板暂无直接预览未打标数据的功能，更多的是日志和任务结果的查阅，或者可以确认是否需跳转到【打标数据看板】）。

## 需求理解

### 功能边界
**包含功能：**
- [ ] 调整 `GetTaskBatches` 的返回值或前端处理逻辑，让任务历史数据中带有明确的“所属数据集”名称。
- [ ] 前端 `TaskKanban.vue` 的“任务历史”表格新增一列“目标数据集”。
- [ ] 前端 `TaskKanban.vue` 的顶部筛选栏增加一个“按数据集过滤”的下拉框。

**明确不包含（Out of Scope）：**
- [ ] 打标引擎底层的隔离重构（已在 Phase 2 完结）。
- [ ] 独立的复杂数据查询与大盘统计（已在 Dashboard 优化中处理）。

## 疑问澄清

### P0级问题（必须澄清）
1. **任务记录的 DTO 映射**
   - 建议方案：前端在获取到 `GetTaskBatches` 列表后，通过调用 `ListDatasets` 获取全量数据集列表，并在前端 computed 中将 `dataset_id` 映射为 `dataset_name`，这符合我们系统目前大量采用的前端组装模式，无需后端大改 DTO。

## 验收标准

### 功能验收
- [ ] “任务历史”列表中，每条任务均能正确显示其所属的“目标数据集”名称。
- [ ] 用户可以通过顶部的下拉筛选器，按特定的数据集过滤任务历史。
- [ ] 创建新任务后，新任务能够正确出现在对应的过滤视图中。
