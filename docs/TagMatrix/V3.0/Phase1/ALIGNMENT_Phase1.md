# 需求对齐文档 - Phase 1: 基础设施与数据源管理重构

## 原始需求
根据 `V3.0_REFACTORING_PLAN.md` 启动 Phase 1: 基础设施与数据源管理重构。
- **后端**：实现 `sys_datasets` 的 CRUD 接口，改造数据导入逻辑（上传时必须选择新建数据集或追加到已有数据集）。
- **前端**：【数据源管理】页面从“单一大网格”重构为“数据集列表视图”，点击数据集后进入对应的专属详情宽表。

## 项目上下文
### 技术栈
- 编程语言：Golang (Wails) / TypeScript (Vue 3 + Element Plus)
- 数据库：SQLite (GORM)

### 现有架构理解
- 当前 `raw_data_records` 无外键约束，包含动态的 JSON 字段。
- 现有的打标逻辑依赖于隐式的 `数据来源` 字段。
- V3.0 的核心是物理隔离，通过 `dataset_id` 连接实体。

## 需求理解

### 功能边界
**包含功能：**
- [x] 后端：新增 `sys_datasets` 数据库模型及 CRUD Wails API。
- [x] 后端：修改 `raw_data_records` 模型，加入 `dataset_id` 外键。
- [x] 后端：调整数据导入解析接口（Excel/CSV），支持在导入时选择或创建 `Dataset`，并解析/更新 `schema_keys`。
- [x] 后端：清空现有的旧数据表（包括 `raw_data_records`, `tag_task_batches`, `sys_match_rules`, `sys_entity_tags`, `tag_task_logs` 等），因为这是一次破坏性重构（根据用户的确认）。
- [x] 前端：`DataSource.vue` 页面重构，使用“左右分栏布局”。左侧是数据集列表，右侧是选定数据集的宽表详情。

**明确不包含（Out of Scope）：**
- [x] 标签与规则配置中心的重构 (Phase 2 的内容)。
- [x] 打标任务引擎与看板的深度改造 (Phase 3 的内容)。
- [x] 导入导出资产解耦 (Phase 4 的内容)。

## 疑问澄清及确认
- **Q1: 数据库旧数据如何处理？**
  - **确认结果**：直接清空旧数据，轻装上阵。这需要我们在后端启动时或者写个迁移脚本去 Drop 所有的相关表或清空记录。最干净的方法是在 `app.go` 的初始化中直接 `db.AutoMigrate` 新表结构，并提供一键清理接口或修改建表逻辑，但这会报错，外键冲突。最好的方法是修改 model 的结构，然后让 GORM 重新迁移，或者我们可以直接删除现有的 sqlite 数据库文件，让它重新创建，既然是开发分支。
- **Q2: 数据源 UI 布局形式？**
  - **确认结果**：左右分栏布局（左侧数据集列表，右侧宽表详情）。

## 验收标准
- [x] 能正常创建、重命名、删除数据集。
- [x] 数据导入时可以关联到特定数据集。
- [x] 导入后的数据在宽表里正确展示其专属的 Schema。
- [x] 不同数据集之间的数据物理隔离。
