# TagMatrix V3.0 核心架构重构总览与实施计划

## 1. 重构背景与愿景

随着 TagMatrix 业务场景的扩展，从“单一文件处理工具”向“通用数据治理中台”演进的过程中，暴露出原有架构的局限性：异构数据源引发的“前端列爆炸”，以及全局打标规则与不同表头产生的“交叉污染”。

V3.0 的核心愿景是：**实现数据源物理隔离，保持业务标签全局统一，将打标规则精准下放到具体数据集**。这标志着 TagMatrix 正式从一个单体工具走向企业级的数据资产平台。

## 2. 核心架构演进路线

V3.0 的重构主要分为两大并行核心：

*   **数据集隔离 (Dataset Isolation)**：彻底废弃单一无限大的 `raw_data_records` 数据池，引入类似 Excel Sheet 的“数据集”概念。每次导入的异构数据独立成表（逻辑上），拥有自己专属的 Schema（表头）。
*   **标签与规则解耦 (Global Tagging, Local Rules)**：标签（Tag Tree）作为全系统的业务资产，保持**全局唯一**；而实现这些标签的判定条件（Rule），则作为工程手段，与具体的**数据集**强绑定。

## 3. 数据库模型变更清单

本次重构需要对底层 SQLite 数据库进行以下升级：

1.  **新建 `sys_datasets` 表**
    *   字段：`id`, `name`, `description`, `schema_keys` (JSON格式的表头字段数组), `created_at`, `updated_at`, `deleted_at`
2.  **改造 `raw_data_records` 表**
    *   新增字段：`dataset_id` (外键关联 `sys_datasets`)
    *   废弃/迁移：将原 JSON 中的“数据来源”等隐式标记，显式转换为 `dataset_id` 物理关联。
3.  **改造 `sys_match_rules` 表**
    *   新增字段：`dataset_id` (必填外键)
    *   逻辑变更：一条规则现在必须同时拥有 `tag_id` (全局标签) 和 `dataset_id` (专属数据集)。
4.  **改造 `tag_task_batches` 表**
    *   新增字段：`dataset_id`
    *   逻辑变更：任务批次必须明确其打标的目标数据集。

## 4. 模块改造实施阶段 (4 Phases)

为了确保系统平稳过渡，V3.0 的开发实施分为四个阶段：

### Phase 1: 基础设施与数据源管理重构
*   **后端**：实现 `sys_datasets` 的 CRUD 接口，改造数据导入逻辑（上传时必须选择新建数据集或追加到已有数据集）。
*   **前端**：【数据源管理】页面从“单一大网格”重构为“数据集列表视图”，点击数据集后进入对应的专属详情宽表。

### Phase 2: 标签与规则配置中心重构 (核心挑战) [x]
*   **后端**：更新 `SaveRule` 和 `GetRuleByTag` 接口，支持按 `dataset_id` 过滤和保存。
*   **前端**：【标签与规则配置】页面左侧保持全局标签树不变。右侧工作台重构为**按数据集分组的规则卡片视图**。新建规则时强制要求先选择目标数据集，并动态加载该数据集的 `schema_keys` 供条件字段下拉框使用。

### Phase 3: 打标任务引擎与看板改造
*   **后端**：Matcher 引擎在执行打标任务时，严格按照传入的 `dataset_id` 去 `sys_match_rules` 中筛选专属规则，并只在归属于该 `dataset_id` 的原始数据中进行运算。
*   **前端**：【打标任务看板】发起打标前，增加“选择目标数据集”的必填项。看板数据展示时，支持按数据集过滤和预览。

### Phase 4: 导入导出资产解耦 (数据中台化)
*   **全局资产导出**：提供纯净的全局标签树导出功能 (`tags.json`)。
*   **局部资产导出**：在数据集管理中，提供“数据集结构 + 专属打标规则”的打包导出功能 (`dataset_with_rules.json`)，实现业务资产与工程代码的物理分离流转。
