# TagMatrix V3.0 全局标签与数据集专属规则解耦重构设计文档

## 1. 背景与核心矛盾

在引入了“数据集隔离”概念后，原有的“标签（Tag）与规则（Rule）强绑定”模式暴露出严重的逻辑缺陷和维护成本：

*   **数据孤岛与重复建设**：如果“标签”必须属于某个特定的“数据集”（即 Local Tagging 方案），那么像“高潜客户”这样的全局业务概念，必须在“订单表”、“用户表”、“活动表”三个不同的数据集里各自创建一个同名标签。这导致后续无法进行全局视角的标签画像和数据聚合。
*   **概念混淆**：标签（Tag）是**业务资产**，代表统一的业务语意；而规则（Rule）是**工程手段**，代表如何从具体的表中提取符合该语意的数据。两者不应强行捆绑在一起。

## 2. 核心重构思想：Global Tagging, Local Rules

为了将 TagMatrix 升级为标准的数据中台架构，必须实现标签体系与规则体系的彻底解耦：

*   **全局标签体系 (Global Tag Tree)**：标签树在系统中是**全局唯一**的，不依附于任何特定的数据集。它是全公司的标准数据字典（例如：电商标签树）。
*   **数据集专属规则 (Local Rules)**：具体的打标规则（匹配条件）是和**具体的数据集**强绑定的。同一个全局标签（如“VIP”），可以有针对“订单数据集”的规则 A（消费 > 1000），也可以有针对“积分数据集”的规则 B（积分 > 5000）。

## 3. 数据库模型变更规划

### 3.1 废弃原有的 `sys_match_rules` (打标规则表) 设计

原有的规则表是一对一挂载在 `sys_tags` 上的（通过 `tag_id`），现在需要升级为“一对多”的挂载关系，并且必须引入 `dataset_id` 的强约束。

*   **表结构更新 `sys_match_rules`**:
    *   `id` (主键)
    *   `tag_id` (关联到全局标签表 `sys_tags.id`)
    *   `dataset_id` (新增：关联到数据集表 `sys_datasets.id`，必填项)
    *   `name` (规则组名称)
    *   `rule_json` (具体的匹配逻辑 JSON)
    *   `is_enabled`, `priority` 等保持不变。
    *   **联合唯一索引**：可以考虑增加 `UNIQUE(tag_id, dataset_id)`（如果业务上限制一个标签在一个数据集上只能有一套规则组），或者不加索引（允许一个标签在一个数据集上有多个规则组，通过优先级 `priority` 区分执行顺序）。

## 4. 核心业务模块改造方案

### 4.1 标签与规则配置 (TagRuleConfig) 页面重构

这是本次解耦重构中 UI 交互变化最大的地方。

*   **左侧区域：全局标签树 (Global Tag Tree)**
    *   依然展示全量的标签层级结构。
    *   **移除**原来直接展示在标签节点上的“是否有规则”的小图标（因为现在规则是分散在不同数据集里的）。
*   **右侧区域：规则配置工作台 (Rule Workspace)**
    *   当点击左侧某个全局标签（如“高潜客户”）时，右侧不再是单一的规则配置表单。
    *   **视图切换**：右侧展示该标签下所有已配置的规则，并按**数据集**进行清晰的分组卡片展示。例如：
        *   `[卡片] 针对【订单数据集】的打标规则 (Rule A)`
        *   `[卡片] 针对【积分数据集】的打标规则 (Rule B)`
    *   **新建规则交互**：
        1.  点击“新建打标规则”按钮。
        2.  **强制第一步**：弹窗或下拉框要求用户选择“**目标数据集**”。
        3.  **动态渲染**：选择数据集后，系统加载该数据集的 Schema（表头），规则配置器里的“条件字段”下拉框仅展示该数据集的专属字段。
        4.  保存时，将 `tag_id`, `dataset_id`, `rule_json` 一并写入 `sys_match_rules`。

### 4.2 打标任务看板 (TaskKanban) 引擎执行逻辑变更

在执行打标任务时，引擎的调度逻辑必须更加精准：

*   用户发起任务时，先选择了**目标数据集**（例如“订单数据集”）。
*   **规则拉取逻辑**：引擎去 `sys_match_rules` 表中，只拉取那些 `dataset_id == 订单数据集ID` 的规则。
*   **引擎执行**：Matcher 引擎严格使用这批专属规则，针对该数据集的 `raw_data_records` 进行运算。忽略其他数据集的规则，彻底避免空跑和错乱。

### 4.3 导入导出体系解耦 (Import/Export Refactoring)

这是为了支持企业级数据资产流转的必经之路。

*   **全局标签体系独立导出 (`tags.json`)**：
    *   提供纯粹的标签树导出，只包含层级、名称、颜色等元数据。
    *   **目的**：使得标准的行业标签字典可以在不同系统或租户间无损迁移，不附带任何底层表结构的包袱。
*   **数据集与规则打包导出 (`dataset_with_rules.json`)**：
    *   在【数据源管理】中提供导出功能。
    *   导出的包包含：该数据集的 Schema（表头定义） + 依赖该数据集的所有 `sys_match_rules`（带上它们关联的全局 Tag 路径/名称，以便导入时能对接到目标系统的全局标签树上）。
    *   **目的**：确保特定业务场景（表结构+判定逻辑）的完整复刻。导入时，如果发现依赖的全局标签不存在，系统可提示自动创建或让用户手动映射。

## 5. 演进路线图与开发建议

1.  **第一阶段**：完成底层的 `sys_datasets` 引入，并将 `raw_data_records` 和 `tag_task_batches` 挂载到数据集上（物理隔离）。
2.  **第二阶段**：执行本设计文档中的“标签与规则解耦”。修改 `sys_match_rules` 结构，重构【标签与规则配置】页面的右侧交互逻辑。
3.  **第三阶段**：重写导入导出逻辑，实现标签元数据与规则工程代码的物理分离。
