# 需求对齐文档 - DatabaseAdmin系统数据库管理

## 原始需求
根据 `DEVELOPMENT_PLAN.md` 和 `UI_DESIGN.md`，需要为 TagMatrix 开发“系统数据库管理”高级功能模块。该模块主要面向高阶用户或系统管理员，提供对底层 SQLite 数据库的全面控制。

## 项目上下文

### 技术栈
- 编程语言：前端 TypeScript/JavaScript，后端 Go
- 框架版本：前端 Vue 3 + Element Plus + Pinia + Vue Router，底层框架 Wails
- 数据库：SQLite (基于 Go 后端)
- 部署环境：Windows 客户端应用 (Wails 构建)

### 现有架构理解
- 架构模式：前后端分离（Wails IPC 通信）
- 核心模块：前端视图层（Vue）、Wails 绑定层（Go methods）、底层服务/数据库操作层
- 集成点：通过 Wails 向前端暴露的 Go Backend APIs 操作数据库和文件系统。

## 需求理解

### 功能边界
**包含功能：**
- [x] 全局设置/Pinia中的“开发者模式”状态管理及导航栏入口动态隐藏。
- [x] **阶段二**：SQL 查询终端，带高亮代码编辑器（如 Monaco / CodeMirror / 或者手写带基本样式的 textarea 以保持简单），执行自定义 SQL，查看结果。
- [x] **阶段三**：可视化表结构与数据管理。实现物理系统表查询，以及基于 JSON 数据集的动态逻辑表展开展示、内联修改（并回写到对应的 JSON 字段）和删改查操作。
- [x] **阶段四**：数据库的快照备份列表、一键备份、高危恢复机制，以及数据库导入导出功能。

**明确不包含（Out of Scope）：**
- [x] 完整的外部数据库连接功能（只管理本应用的内置 SQLite 数据库）。
- [x] 复杂的权限角色管理体系（基于当前的开发者模式开关即可）。

## 疑问澄清

### P0级问题（必须澄清）
1. **代码编辑器依赖引入**：
   - 背景：第二阶段要求“带语法高亮的 SQL 编辑器”。目前 `package.json` 未安装 Monaco 或 CodeMirror。
   - 影响：引入 Monaco 可能会显著增加应用打包体积，而且配置相对繁琐。如果为了轻量，使用简易高亮或普通的多行文本框是否可接受？
   - 建议方案：优先引入轻量级的 `codemirror` 或简单的 vue 代码编辑器组件；若不想增加第三方依赖，初期可使用原生的 `<el-input type="textarea">`。请问如何抉择？

2. **数据库文件锁定机制**：
   - 背景：在使用覆盖数据库或者从快照恢复时，Go 底层的 SQLite 连接如果不关闭会导致文件被锁。
   - 影响：可能导致备份恢复失败。
   - 建议方案：在 Go 端提供一个专门用于“释放 DB 连接、执行替换、重新建立连接”的接口。恢复后前端需重新加载整个应用。

## 验收标准

### 功能验收
- [ ] 只有开启开发者模式才能在导航栏看到入口。
- [ ] 能在前端直接执行 `SELECT` 语句并呈现结果。
- [ ] 可以在 JSON 解析模式下，看到 `raw_data_records` 中的业务数据字段，并能像操作普通表格一样编辑修改。
- [ ] 备份中心能够正常创建快照、恢复并要求应用重载。

### 质量验收
- [ ] 对高危操作有二次确认弹窗或红色警示。
- [ ] 修改虚拟 JSON 字段时，不会误删记录中其他未修改的 JSON 数据。
