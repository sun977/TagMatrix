# TagMatrix V3.0 数据集隔离架构重构设计文档

## 1. 背景与痛点
当前 TagMatrix 系统将所有导入的数据统一存放在 `raw_data_records` 表的动态 JSON 列中。虽然天然兼容异构数据，但随着多次导入不同表头（Schema）的文件，暴露出了以下严重问题：
- **展示层“列爆炸”**：前端合并了所有导入文件的字段，导致出现大量空值的稀疏矩阵。
- **规则配置易错**：用户配置规则时无法精准选择对应字段，只能手动盲猜或容易拼写错误。
- **打标引擎无意义消耗**：针对 A 表的规则可能在 B 表数据上空跑，引发逻辑交叉污染和潜在的异常。

## 2. 核心重构思想：引入“数据集 (Dataset)”
借鉴成熟 BI 平台与多维表格的设计，将原来的“全局无限大数据池”改造为“多数据集（类似 Excel 的多个 Sheet）”管理模式。
- 每个异构文件（如“用户表”、“订单表”）作为一个独立的**数据集**。
- 每个数据集独立维护自己的 Schema（表头结构）。
- 打标规则和打标任务严格与特定数据集绑定，实现物理和逻辑的双重隔离。

## 3. 数据库模型变更规划
为了支持数据集的概念，底层关系型数据库（SQLite + GORM）需要进行以下重构：

### 3.1 新增表 `sys_datasets` (数据集管理表)
此表用于记录所有被创建的数据集及其 Schema 信息。
- `id` (主键)
- `name` (数据集名称，如“2026年商品库”)
- `description` (描述)
- `schema_keys` (JSON 格式，记录该数据集包含的表头字段数组)
- `created_at`, `updated_at`, `deleted_at`

### 3.2 修改 `raw_data_records` (原始数据表)
原始数据必须归属于某个具体的数据集。
- 新增字段 `dataset_id`，作为外键关联到 `sys_datasets`。
- 废弃原有的隐式“数据来源”字段区分，改为明确的数据集归属。

### 3.3 修改 `sys_match_rules` (打标规则表)
规则必须被限定在特定的表头结构上。
- 新增字段 `dataset_id`，标识该规则（或规则组）仅在特定数据集上生效。

### 3.4 修改 `tag_task_batches` (任务批次表)
打标任务执行的目标必须是确定的。
- 新增字段 `dataset_id`，标识该次打标任务针对的数据集。

## 4. 核心业务模块改造方案

### 4.1 数据源管理 (DataSource)
- **UI 变更**：由原来的“单表大网格”改为“数据集列表”。点击某个数据集后，进入该数据集的详情页（展示其专属宽表）。
- **导入逻辑**：导入 Excel/CSV 时，提供“新建数据集”或“追加到已有数据集”的选项。若选择追加，后端需校验文件的表头是否与该数据集的 `schema_keys` 兼容。

### 4.2 标签与规则配置 (TagRuleConfig)
- **UI 变更**：创建或编辑规则组时，增加“关联数据集”的必填下拉选项。
- **交互增强**：一旦选择了数据集，规则配置里的“条件字段”下拉框将精准加载该数据集的 `schema_keys`，杜绝手动输入错误，大幅提升配置体验。

### 4.3 打标任务看板 (TaskKanban)
- **发起任务流程**：
  1. 选择目标数据集。
  2. 选择该数据集关联的专属规则（级联过滤）。
  3. 设定打标策略和打标模式（追加/覆盖）。
  4. 执行打标。
- **引擎执行**：Matcher 引擎在查询数据时，严格根据 `dataset_id` 过滤 `raw_data_records`，确保绝对的作用域安全，防止张冠李戴。

## 5. 跨表打标签的远期展望 (Cross-table Tagging Roadmap)
目前 V3.0 专注于单表的数据集隔离。针对未来可能出现的“跨表打标签”（如根据订单表给用户表打标）需求，系统架构保持以下演进方向：
- **现阶段推荐**：在 TagMatrix 外部通过 ETL 工具或 Excel VLOOKUP 将多表 Join 成大宽表后再导入系统。
- **远期规划 (V4.0+)**：未来可考虑在 SQLite 层面引入“虚拟视图 (Virtual View)”，允许高级用户编写 SQL 关联多表，将其映射为一个只读的“虚拟数据集”，从而完美复用现有的单表规则配置和打标逻辑。
