# TagMatrix 规则优先级 (Rule Priority) 设计方案与应用场景

## 1. 背景与初衷
在 `SysMatchRule`（自动打标规则表）的设计中，保留了 `priority`（优先级，整型）字段。在复杂的智能打标系统中，随着规则数量的激增、业务逻辑的复杂化，规则之间必然会产生**冲突、重叠或依赖**。优先级字段正是解决这些问题的关键，也是系统从“简单规则过滤”走向“复杂推理引擎”的重要基石。

本文档旨在梳理 `priority` 字段在未来 TagMatrix 演进中的核心应用场景与落地方案，作为后续架构升级的设计参考。

---

## 2. 核心应用场景与落地方案

### 方案一：同组/互斥标签的冲突仲裁（最常见场景）
**场景描述**：
很多时候，同一维度的标签是互斥的。例如“用户价值”维度下有【高价值】、【中价值】、【低价值】。一条数据可能同时命中了“最近登录<3天（高价值）”和“消费金额<10元（低价值）”两条规则。

**优先级用法**：
* 为规则设定优先级，比如“消费金额规则”优先级设为 100，“登录活跃规则”优先级设为 50（数字越大优先级越高）。
* **引擎侧逻辑**：在打标引擎（`TaskEngineService`）评估时，如果发现同一数据命中了同一个互斥标签组下的多个规则，**只保留优先级最高的规则所对应的标签**，直接丢弃低优先级的匹配结果。
* **业务价值**：解决业务上“到底听哪个规则的”这种冲突问题，保证数据的准确性和唯一性。

### 方案二：打标引擎的短路计算与性能优化
**场景描述**：
当系统内有成千上万条规则时，每条数据都要过一遍所有规则（哪怕使用 AST 解析也会有性能损耗），尤其是涉及到正则匹配或 AI 语义理解的复杂规则。

**优先级用法**：
* 将规则分为“高频/简单规则”（如：`status == 1`，优先级设为 90）和“低频/复杂规则”（如正则匹配、大模型调用，优先级设为 10）。
* **引擎侧逻辑**：在 Worker 执行打标前，先根据 `priority` 降序排列所有规则。如果业务允许“命中即停止”（比如黑名单过滤场景，命中一条黑名单规则即可），引擎在执行完高优先级规则并命中后，直接**短路（Break）**，跳过后续低优先级规则的计算。
* **业务价值**：极大降低 CPU 消耗，提升海量数据批处理任务的执行速度。

### 方案三：决定“主副标签（IsPrimary）”的归属
**场景描述**：
当前的 `SysEntityTag` 表中已经设计了 `is_primary`（是否为主标签）字段。一条数据可能会被打上十几个标签，但在前端概览或画像中，只能高亮展示 1-2 个最核心的特征。

**优先级用法**：
* 不同标签的规则设定不同的优先级。例如“涉诈黑产”规则优先级 99，“活跃用户”规则优先级 20。
* **引擎侧逻辑**：数据跑完所有规则后，按命中规则的优先级对打上的标签进行排序。优先级 Top 1（或 Top N）的标签，其 `sys_entity_tags.is_primary` 被置为 `true`。
* **业务价值**：自动为数据提取“核心特征”，前端画像展示时可以突出高优标签，提升业务人员的筛选效率。

### 方案四：多阶段（Pipeline）依赖打标
**场景描述**：
有些高级标签的计算依赖于基础标签。例如，要打上【高潜力客户】标签，前提是该数据必须已经被打上了【已实名】和【活跃】标签。

**优先级用法**：
* 此时 `priority` 退化为**执行阶段（Stage / Phase）**。
* 优先级 100 的规则：基础清洗和初级打标（如提取地域、性别）。
* 优先级 50 的规则：组合打标（如果包含“北京”+“女性”标签，则打上“北京都市丽人”标签）。
* **引擎侧逻辑**：引擎分批次执行。先拉取 `priority >= 100` 的规则跑一轮，将结果写入内存或 DB；然后再拉取 `priority < 100` 的规则，基于上一轮的结果进行二次推理打标。
* **业务价值**：让系统具备实现“复杂图谱（DAG）推理”和“规则级联”的能力。

## 3. 后续落地建议

1. **底层引擎（保留排序预留口）**：
   在 `taskengine_service.go` 加载规则到 Worker 之前，可以先加一行代码把传入的 `rules` 按 `priority` 降序排个序。不管现在有没有短路逻辑，先保证高优先级的规则**先执行**。

2. **前端交互（弱化数字，强化可视化拖拽）**：
   未来在前端配置规则时，尽量不要让用户手动输入 `99`、`10` 这样的数字（用户对数字的边界感知较弱，容易配置混乱）。更好的设计是在前端规则列表支持**拖拽排序**，拖拽后由后端自动计算并更新 `priority` 字段的数值。

3. **当前阶段（保持现状）**：
   在明确具体需要哪种冲突解决策略之前，无需对现有打标核心逻辑做重构。保留好这个字段，它是系统未来向复杂规则引擎演进的关键伏笔。
