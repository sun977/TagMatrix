# 需求梳理
这是一个给数据打标签的系统，用户可以自定义标签和标签属性，可以定义标签的打标规则，然后系统根据标签和打标规则匹配数据，将数据打上对应的标签。
系统有UI界面，可以图形化操作
预留AI能力接口，让系统可以调用AI模型进行分析辅助用户打标（或者代替打标），AI的协议至少需要支持openai协议

## 自定义部分
- 标签名称
- 标签属性（说明）
- 打标签的规则（包含数据匹配规则：先匹配数据在打标签）
- 数据源支持 excel，csv，数据库等

## 系统自动化部分
- 自动匹配数据
- 自动打标签
- 自动记录打标签日志

## 输入输出
- 输入：数据源（excel，csv，数据库等）
- 输出：打标签后的数据（excel，csv，数据库等）
- 日志：打标签日志（包含匹配数据，打标签，记录打标签日志）


## 需求演进
- 系统打标有三种模式，分别是单标签模式，多标签模式，和混合模式
1. 单标签模式：用户只能给数据打上一个标签
2. 多标签模式：用户可以给数据打上多个标签
3. 混合模式：用户可以给数据打上一个主标签，和多个次标签

- 原始数据是多字段的，需要可以指定对一个或者多个字段的内容进行匹配，匹配成功后打标
- AI功能让用户可以和AI模型对话，AI可以帮助用户在数据表中查看或者筛选数据（甚至分析数据，生成SQL语句等），AI药可以告诉用户这个软件工具怎么用
- 数据库目前是sqlite，后续可以考虑支持mysql，postgresql等
- 除了数据支持导入导出以外，标签结构数据，匹配规则数据，打标规则数据也要支持导入导出，格式csv和json
- 用户触发打标流程后，每条记录的打标过程会被记录到日志中（日志包含打标理由，即触发了那条匹配规则，打标规则，计算方式才最终打标结果），用户可以查看打标日志，了解每条记录的打标过程，判断打标是否合理。
- 每轮打标任务要有任务编号或者记录，方便用户如果觉得当前的打标结果不满意，可以回退到之前任意版本的打标结果。
- 每个匹配计算的算子在用户设置的时候可以试运行向用户展示结果，用户可以根据结果调整算子参数，直到得到满意的结果。防止用户设置的算子参数错误导致的打标结果错误或算子不生效的情况。

### 打标数据看板需求演进
- 打标数据看板看到的数据是打标之后的结果，用户可以指定是那一次打标任务结束的结果。
- 打标数据看板用户可以指定展示的字段，也可以展示所有字段。这些字段根据原始数据而来。
- 打标数据看板用户可以根据展示的字段进行筛选，也可以根据所有字段进行筛选。
- 打标数据看板用户可以看到每条数据打标过程的记录，包括匹配规则，打标规则，计算方式，最终打标结果。用户可以根据记录判断打标是否合理。
- 系统内字段使用前嘴 TagM_ 防止和用户数据字段冲突。

## 概念澄清
- 开发模式 (Development Mode / Dev Build) :
    - 含义 : 这是一个 运行环境 的概念。当您在终端输入 wails dev 时，应用就处于开发模式。此时前端具有热重载（Hot Reload），代码未被编译压缩。
    - 行为 : 数据库（ data.db ）和日志文件（ app.log ）会直接存放在当前项目的根目录 . ，方便您用代码编辑器随时查看和修改。
- 调试模式 (Debug Mode) :
    - 含义 : 这是一个 应用功能 概念。它是提供给终端用户的开关（通过【全局设置】界面控制，对应代码中的 config.Adv.DebugMode ）。
    - 行为 : 无论应用是开发状态还是已经打包成 .app/.exe （生产环境），只要开启此开关，系统内部的 Logger 就会提升到 DebugLevel （开启记录所有详细 API 请求），并且底层 GORM 也会记录 每一条 SQL 语句及耗时（原本只记录耗时>200ms的慢查询）。方便排查线上疑难杂症。
- 开发者模式 (Developer Mode) :
    - 含义 : 这是一个 隐藏的高级权限 概念。在【全局设置】中连续点击 5 次“开发者模式”的文字才会解锁这个开关。
    - 行为 : 开启后，会在系统左侧导航栏显示【高级与系统】菜单（即您之前看到的【系统数据库管理】等界面），允许高级用户直接修改系统内部表（物理表/业务表）。
